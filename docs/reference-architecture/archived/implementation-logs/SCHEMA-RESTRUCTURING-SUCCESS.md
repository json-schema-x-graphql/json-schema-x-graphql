## Comparison: Before vs After

### Before (Document-Oriented)

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "title": "Enhanced Government Contract...",
  "properties": {
    "systemMetadata": {
      "type": "object",
      "properties": {
        "systemChain": {
          "items": {
            "type": "object",
            "properties": {
              "dataQuality": {
                "type": "object",
                "properties": { ... }
              }
            }
          }
        }
      }
    }
  }
}
```

**Issues**:

- ❌ No named types
- ❌ Nested objects inline
- ❌ Not reusable
- ❌ typeconv can't process it

### After (Definitions-Based)

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "https://github.com/GSA-TTS/enterprise-schema-unification/schemas/contract-v2",
  "definitions": {
    "DataQuality": {
      "type": "object",
      "description": "Data quality metrics...",
      "properties": { ... }
    },
    "SystemChainEntry": {
      "type": "object",
      "properties": {
        "dataQuality": { "$ref": "#/definitions/DataQuality" }
      }
    },
    "Contract": { ... }
  },
  "$ref": "#/definitions/Contract"
}
```

**Benefits**:

- ✅ Named, reusable types
- ✅ Clean composition with `$ref`
- ✅ Works with standard tools
- ✅ Better documentation

Once validated, you can replace the custom transformation scripts:

#### Option A: Use typeconv CLI (Simplest)

**Update `package.json`**:

```json
{
  "scripts": {
    "generate:graphql-from-json": "typeconv -f jsc -t gql -o generated-schemas/schema_unification.from-json.graphql src/data/schema_unification.schema.json",
    "generate:json-from-graphql": "typeconv -f gql -t jsc -o generated-schemas/schema_unification.from-graphql.json src/data/schema_unification.graphql"
  }
}
```

**Delete** (or archive):

- `scripts/generate-graphql-from-json-schema.mjs` (250 lines)
- Most of `scripts/json-to-graphql.config.mjs` (597 lines)

**Keep** (for custom enums and naming):

- Minimal post-processing script (~50 lines) if needed

#### Option B: Use core-types API (More Control)

Create `scripts/generate-schema-with-core-types.mjs`:

```javascript
#!/usr/bin/env node
import fs from "fs/promises";
import {
  convertJsonSchemaToCoreTypes,
  convertCoreTypesToJsonSchema,
} from "core-types-json-schema";
import {
  convertCoreTypesToGraphQL,
  convertGraphQLToCoreTypes,
} from "core-types-graphql";

// JSON Schema → GraphQL
const jsonSchema = JSON.parse(
  await fs.readFile("src/data/schema_unification.schema.json", "utf8"),
);
const coreTypes = convertJsonSchemaToCoreTypes(jsonSchema);
const graphql = convertCoreTypesToGraphQL(coreTypes);
await fs.writeFile(
  "generated-schemas/schema_unification.from-json.graphql",
  graphql,
);

// GraphQL → JSON Schema
const graphqlSdl = await fs.readFile(
  "src/data/schema_unification.graphql",
  "utf8",
);
const coreTypesFromGql = convertGraphQLToCoreTypes(graphqlSdl);
const jsonSchemaFromGql = convertCoreTypesToJsonSchema(coreTypesFromGql);
await fs.writeFile(
  "generated-schemas/schema_unification.from-graphql.json",
  JSON.stringify(jsonSchemaFromGql, null, 2),
);
```

## Known Limitations & Workarounds

### 1. Enum Type Mapping

**Issue**: typeconv maps JSON Schema enums to GraphQL `String` type

**Current**:

```graphql
type Contact {
  role: String # Should be ContactRole enum
}
```

**Workaround**: Add post-processing to convert specific strings to enums:

```javascript
const graphql = convertCoreTypesToGraphQL(coreTypes);
// Post-process to add enum definitions
const withEnums =
  graphql.replace(/role: String/g, "role: ContactRole") +
  "\n\nenum ContactRole {\n  PRIMARY\n  TECHNICAL\n  ADMINISTRATIVE\n  CONTRACTING_OFFICER\n}";
```

### 2. `value: {}` Fields (Any Type)

**Warnings**: "Type 'any' not supported" (3 instances)

**Affected Fields**:

- `Contract DataExtension.value`
- `AssistExtension.value`
- `EasiExtension.value`

**Workaround**: Define explicit type or use GraphQL `JSON` scalar:

```json
{
  "value": {
    "oneOf": [
      { "type": "string" },
      { "type": "number" },
      { "type": "boolean" },
      { "type": "object" }
    ]
  }
}
```

### 3. Custom Type Names

typeconv uses the definition key names directly. If you need different names in GraphQL, use a mapping config or post-processing.
